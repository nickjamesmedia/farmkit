# Plans

## Active plan: V0.1 beta backend schema + demo seeds
**Status:** Completed  
**Owner:** AI-led, human-approved  
**Started:** 2026-01-11

### Goal
Define the v0.1 beta SQL schema and seed data for Supabase (backend only), aligned with the modular data model and ready for future expansion.

### Non-goals
- Frontend/UI updates or workflow changes.
- New modules beyond v0.1 core (equipment, buildings/containers, maintenance).
- RLS policy authoring or auth flow changes in this plan (tracked separately).
- Production migrations or data backfills.

### Assumptions
- `docs/dev/data_model.md` is the baseline for the v0.1 table set and relationships.
- `supabase/schema.sql` and `supabase/seed_sample_data.sql` are the target files for schema + seed data (unless a new seed file is agreed).
- Seed data should include Canadianisms and cover each account type/role combination needed for testing.
- Only backend artifacts are in scope for this plan.
- V0.1 launch includes Equipment, Buildings, and Maintenance tracking capabilities.
- Seed data will only target app tables; users are created via Supabase Auth.
- New seed file will be `supabase/seed_v0_1_demo.sql`, with the existing seed file archived.

### Work items (ordered)
1) Inventory current prototype schema + data model notes; list required v0.1 tables and relationships.
2) Define the v0.1 SQL schema (tables, constraints, indexes, enums), calling out deltas from prototype.
3) Document migration/rollback approach for schema changes (required before edits).
4) Specify demo seed requirements: farms (parent + child), modules, users/profiles, memberships, roles, people, equipment, containers/buildings, maintenance logs (and any supporting tables).
5) Draft seed data set (Canadian flavor) and identify test personas for each account type/role.
6) Implement schema updates in `supabase/schema.sql`.
7) Archive `supabase/seed_sample_data.sql` and create `supabase/seed_v0_1_demo.sql`.
8) Implement seed SQL in `supabase/seed_v0_1_demo.sql`.
9) Validate by inspection and note any missing coverage or open questions.

### Schema deltas (prototype -> v0.1)
- Replace `app_users` with `user_profiles` + `farm_memberships`.
- Replace `locations` + `buildings` with `containers` + `building_details`.
- Add `farms.parent_farm_id` and `farm_details` for site/contact fields.
- Add `roles`, `modules`, and `farm_modules` for access tiers + modular toggles.
- Add `people` for shared-login attribution in maintenance logs.

### Migration / rollback approach (draft)
- Dev reset: drop/recreate the schema from `supabase/schema.sql` (prototype data is disposable).
- Preserve old data (if needed): export prototype tables before applying the v0.1 schema.
- Rollback: reapply the previous `supabase/schema.sql` and restore data from the export.

### Acceptance criteria
- A complete v0.1 SQL schema is defined and ready for implementation in Supabase.
- Seed plan covers every v0.1 table with sample data and includes all account types.
- Canadian-themed demo data is specified for the dev Supabase environment.
- `.agent/TASKS.md` reflects current status and next steps.

### Validation
- Schema and seed files reviewed for completeness and consistency with `docs/dev/data_model.md`.
- No secrets or internal customer names added to seed data.
