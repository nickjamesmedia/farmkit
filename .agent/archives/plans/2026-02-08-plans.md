# Plans

## Active plan: v0.1 beta readiness (modules + RLS + frontend scoping)
**Status:** In progress (pending manual verification + Supabase apply)  
**Owner:** Codex + Human  
**Started:** 2026-02-01

### Goal
- Get ready for v0.1 beta launch with access control enforced by RLS and frontend behavior (scope, filters, module UI) aligned with what RLS allows.
- Implement the small fix: on `/maintenance/add`, allow setting `maintenance_logs.status` on create (`open`/`closed`, default `open`).

### Non-goals
- Implementing v0.2 modules (Fields, Livestock, Storage & Bins) - document only.
- UI redesign (keep current layout/theme; only hide/show existing elements where needed).
- Treating frontend filtering as an access-control boundary (RLS remains the boundary).
- Making the dev-only RLS SQL production-ready (separate effort).

### Assumptions
- Backend enforcement is Supabase Postgres RLS (`auth.uid()`), not UI gating.
- Each `farm` is a physical location/site.
- Farm hierarchy is intentionally two-level for now: parent farm -> child farms; child farms cannot have children.
- Module toggles are set at the parent farm and inherited by child farms (no child overrides).
- ERP (emergency response planning) info should be broadly readable to users for locations they work at (including parent ERP info for child-farm members).

## Canonical intentions (from 2026-02-01 discussion)

### Modules
- Features are grouped into modules.
- When a new module is introduced later, it should be disabled by default for existing farms.
- Modules are configured at the parent farm and inherited by all child farms/members.
- Disabling a module should:
  - hide the module's data via RLS, and
  - hide related UI elements (links/buttons/routes) for UX,
  - without breaking other modules (graceful degradation).
- Some module gating will be:
  - per table (all rows in a table), and/or
  - per row (e.g., `containers.container_kind`), and/or
  - nested modules (module families).

### Roles & membership inheritance (directional target for v1, influences v0.1)
- Admin:
  - assigned at the parent farm
  - admin access across the parent farm and all child farms
- Manager:
  - assigned to a farm
  - can manage that farm and any child farms
- User:
  - assigned to a farm
  - day-to-day access for that farm and any child farms (primarily submitting logs)
  - can read operational data broadly, excluding personal/admin-sensitive data
  - must be able to see basic parent farm info (contact + ERP) for locations they work at

### Farm hierarchy
- Two levels only for now:
  - Level 1: parent/admin farm
  - Level 2: child farms (locations)
  - No child-of-child (no grandchildren)
- Users/managers may have memberships in multiple parent farms (separate tenants).

### ERP (Emergency Response Planning)
- Location-related items should expose ERP info.
- ERP info should remain broadly accessible to users for any farm location they might be working at.

### v0.1 note: admin child-farm enrollment (explicit memberships)
- Even though RLS can treat parent-farm Admin as having effective access to child farms, we want Admins to appear as members of child farms in the UI and in membership lists.
- We will implement a database helper + triggers to auto-enroll parent-farm Admins into all child farms, marking those rows as inherited so they can be revoked cleanly on demotion/removal.

## Current implementation (reference) and key gaps

Reference files:
- Schema: `supabase/schema.sql`
- Dev-only RLS policies: `supabase/rls_policies_dev.sql`

### Current RLS helper behavior (important)
- `farmkit_has_farm_access(target_farm_id)` is currently symmetric up/down for parent/child.
- `farmkit_module_enabled(target_farm_id, module_key)` currently checks `farm_modules` at that exact `farm_id` (no inheritance).
- `containers` and `building_details` are currently gated by the `buildings` module as a whole-table gate.

### Gaps vs the canonical intent
1) **Module inheritance is not enforced**
   - Canonical: parent farm sets module toggles; children inherit.
   - Current: child rows are gated by child `farm_modules` rows; turning a module off at the parent does not necessarily turn it off for the child.

2) **Child membership currently implies access to parent module data**
   - Canonical: child members need parent farm metadata/ERP, but not necessarily parent module data.
   - Current: `farmkit_has_farm_access` is used by module tables and grants child->parent access broadly.

3) **Containers will need row-level module gating**
   - Canonical: Containers will host Buildings and Storage & Bins, which must be independently disableable.
   - Current: gating `containers` by a single `buildings` module would break Storage later.

4) **Frontend module UI gating exists only as an intention, not a consistent pattern**
   - We should hide UI elements for disabled modules, but keep the style/layout similar.

## Decisions to confirm before implementation (blocking)

### D1) Module keys and nesting strategy (schema + RLS + UI labels)
We need a decision on how v0.1 "Containers" and "ERP" map to `modules.key`.

Options:
- **Option A (minimal churn for v0.1):** keep existing keys `equipment`, `maintenance`, `buildings` and treat `buildings` as "Containers > Buildings" in UI copy and docs. Defer a dedicated `containers` module key until Storage exists.
- **Option B (align naming now):** introduce `containers` (base), `containers_buildings` (sub), and `erp` module keys now; migrate references to the new keys.

**Decision (confirmed): Option B.**

### D2) Effective module owner farm (inheritance)
Confirmed: module configuration is sourced from the parent farm and inherited by child farms (no child overrides yet).

Constraint: do not design this in a way that prevents adding child overrides later; prefer an "effective module config farm" helper that can evolve.

### D3) ERP module behavior
**Decision (confirmed):** ERP is default-enabled but can be disabled by farms.

Also confirmed: all v0.1 modules are default-enabled but toggleable by farms.

### D4) Default behavior when a `farm_modules` row is missing
**Decision (confirmed):** missing `farm_modules` row => fallback to `modules.default_enabled` (requires schema change).

### D5) Child members: parent metadata vs parent module data
**Decision (confirmed):** child-farm members can read parent `farms` + `farm_details` (ERP/contact), but cannot read parent module tables (equipment/containers/maintenance/etc).

## Proposed changes (for confirmation)

### Proposed RLS structure changes
1) **Split farm access helpers into two concepts**
   - `farmkit_can_access_farm_data(target_farm_id)` (module data)
     - true when the user is an active member of the target farm OR the target's parent
     - (downward inheritance only)
   - `farmkit_can_read_farm_metadata(target_farm_id)` (metadata + ERP)
     - true when the user is an active member of:
       - the target farm, OR
       - the target's parent, OR
       - a child farm of the target
     - (allows child members to read parent metadata)

2) **Update policies to use the correct helper**
   - `farms` and `farm_details` SELECT => metadata helper
   - module tables (equipment/containers/people/maintenance_logs/etc) SELECT => farm data helper

3) **Implement module inheritance in `farmkit_module_enabled`**
   - Add helper: `farmkit_module_owner_farm_id(target_farm_id)` returning:
     - `parent_farm_id` if present, else `target_farm_id`
   - Change `farmkit_module_enabled` to check `farm_modules` for the owner farm id.

4) **Row-level gating for Containers submodules (to prevent cross-module breakage)**
   - Base Containers module should gate all `containers` rows.
   - Buildings submodule should gate only:
     - `containers` rows where `container_kind = 'building'`, and
     - `building_details`.
   - Storage & Bins submodule is v0.2 (documented only) but design gating now so adding it doesn't force a rewrite.

5) **Restrict module writes to parent farms**
   - Prevent editing module toggles for child farms (since modules are parent-configured).

### Proposed schema changes
1) **Enforce two-level farm hierarchy**
   - Add a constraint/trigger preventing a farm whose parent is itself a child (no grandchildren).

2) **Module catalog changes (D1 confirmed Option B)**
   - Add module keys: `containers`, `containers_buildings`, `erp` (and reserve `containers_storage` for v0.2).
   - Update RLS module gating to use these keys.
   - Update Farm Setup module UI to display nested modules (Containers -> Buildings, Storage & Bins later) without changing overall layout/theme.

3) **Module defaults (D4 confirmed)**
   - Add `modules.default_enabled boolean not null` (default true for v0.1 modules).
   - When new modules are introduced later, set `modules.default_enabled = false` to keep them disabled by default for existing farms until explicitly enabled.
   - Update `farmkit_module_enabled(...)` to fallback to `modules.default_enabled` when there is no `farm_modules` row.
