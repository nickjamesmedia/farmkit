## Active plan: RLS setup plan for v0.1 data model
**Status:** Completed  
**Owner:** Codex  
**Started:** 2026-01-18
**Completed:** 2026-01-18

### Goal
Define the RLS policy approach for the v0.1 data model, document it in a dev planning doc, add a dev-only RLS audit page for testing visibility, and draft dev-only RLS SQL policies.

### Non-goals
- Implementing or deploying RLS SQL policies.
- Changing the data model or table structure.
- Broad UI redesigns or unrelated feature work (only dev RLS audit page + version badge link + `farmkit` branding updates).
- Shipping production-ready migrations (dev-only policies for now).

### Assumptions
- Supabase Postgres + auth.uid() is the enforcement layer.
- Access is scoped by active `farm_memberships` rows.
- Initial roles remain admin/manager/user with no custom roles yet.
- The dev RLS audit page is behind existing auth and is not linked in primary nav.
- RLS SQL will live in a new dev-only file until v0.1 beta.

### Work items (ordered)
1) Review `docs/dev/data_model.md` for the v0.1 table list and scope.
2) Define baseline RLS patterns (farm-scoped, membership-scoped, system tables).
3) Draft a per-table access matrix with role expectations and open questions.
4) Write the RLS planning doc in `docs/dev/rls_plan.md`.
5) Add a dev-only RLS audit page that lists visible records for selected tables.
6) Add a route for the dev page behind `RequireAuth`.
7) Make the version badge link to the dev page and update text to `farmkit`.
8) Draft a dev-only RLS SQL file with policies + rollback notes.
9) Update `.agent/TASKS.md` with status and next steps.

### Acceptance criteria
- `docs/dev/rls_plan.md` captures scope, patterns, and per-table access expectations for v0.1.
- Open questions and decisions needed are listed explicitly.
- The dev RLS audit page renders a log-style table of visible records.
- Clicking the version badge navigates to the dev page.
- The version badge and document title use `farmkit` (lowercase).
- A dev-only RLS SQL file exists with enable/disable guidance and rollback notes.

### Validation
- Manual review for consistency with `docs/dev/data_model.md` and `.agent/CONTEXT.md`.
- Manual navigation to the dev page and a quick scan for errors.
- User reported quick high-level RLS checks in dev looked good.
